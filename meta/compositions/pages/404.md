## 0. Preamble — why this composition exists at all

The spec's §9 explicitly marks the designer composition as **optional**: "The composition lift is small enough that a dedicated composition doc may not be earned — at the designer's call. If the designer authors one, it lands at `meta/compositions/pages/404.md`. If not, the engineer composes directly from this spec + the `/principles` end-CTA pattern + the home Hero precedent."

This composition is authored anyway, for two reasons:

1. **Consistency.** The four canonical routes (`/`, `/why-ai`, `/roles`, `/principles`) and `/about` v2 each carry a composition doc. `/404` is the fifth surface in the site's compositional vocabulary; a missing recipe here is a future-audit gap that would re-surface as a retroactive ratification doc (cf. the home composition's §0 "after-the-fact" preamble). Authoring inline is cheaper than authoring retroactively.
2. **Lock the small decisions before they drift.** The spec leaves four composition-layer calls open: `<Hero size>`, `<Hero variant>`, `<Hero entrance>`, and vertical positioning. Each is a one-line decision but each carries brand-register weight. Locking them here means the engineer composes against an Arian-ratified recipe rather than re-deriving from the spec under build pressure.

## 1. Intent

`/404` should feel like the site it salvages you back into. A visitor lands here by mistyping or following a stale link; the page's job is to make the routing failure legible without degrading register, then hand the visitor back through the existing nav, a single home-link CTA, or — after the rev-2 override — a short editorial list of important routes with one-line descriptors. The reader's eye lands on one short declarative line, registers the dead end in under five seconds, scans a single-sentence resolution path, decides whether to click home (Hero CTA) or to scan the descriptor list (new unit), and acts. Density is still deliberately low: one Hero block on `--bg`, a quiet sub-footer-register list below, generous breathing room, the SiteShell chrome doing the routing work above and below. The page is still one viewport tall at every desktop height — see §3 math.

The visual posture is **calm, not contrite**. An operator-first brand whose error page performs remorse undercuts its own restraint. The reader should register three things in sequence: (a) "I'm somewhere wrong"; (b) "this brand still feels like itself here"; (c) "here are the places worth going next, with reasons." The third register is the rev-2 addition. The list earns its place only if it reads as *editorial supplement* (a quiet "here's where to actually go" register, descriptor-led, one short clause per row), not as *nav duplication* (label-only, button-row, card-grid). The composition's typographic call (Section 3 — new) is what makes or breaks that register.

## 2. Section-by-section composition

The spec's §4 IA listed three units; the rev-2 override adds a fourth. Total render order: **4 units** — `<SiteShell>` top + `<Hero>` body + **new body link list** + `<SiteShell>` footer. Total body content target: ≤75 words across Hero (≤50) and link list (3 rows × ≤6 words ≈ ≤18 words). Single viewport tall at typical desktop heights; the page does not scroll on a 1440×900 viewport. Mobile may scroll slightly to reach the list — see §3 math.

### Section 1 — `<SiteShell>` (page chrome)

- **DS primitive(s)**: `<SiteShell>` (organism) from `@poukai-inc/ui@0.9.0`. Wrapped site-side by the existing `ShellWrapper.tsx` pattern (substance carrier across the React boundary; see home composition §2). No `client:*` directive — renders as static HTML at build time.
- **Props (substantive)**:
  ```
  <SiteShell
    currentRoute={null}                       // per spec §9: no nav item marked current on /404
    routes={[
      { href: "/why-ai",     label: "Why AI" },
      { href: "/roles",      label: "Roles" },
      { href: "/principles", label: "Principles" },
      { href: "/about",      label: "About" },
    ]}                                        // four-item nav, funnel order, matches post-/about-v2 state
    footer={<p>© <year> pouk.ai · <a href="mailto:hello@pouk.ai">hello@pouk.ai</a></p>}
  >
    {/* Hero — Section 2 below */}
  </SiteShell>
  ```
- **Layout / spacing**: `<SiteShell>` owns its own header/footer chrome internally. The DS handles `--page-pad` on the outer edge, the wordmark height (56px per ADR-0008), and the nav gap. The page-content wrapper between header and footer is `.site-page` in `site.css`, which applies `max-width: var(--content-max)` (64rem) and `padding-block: var(--space-12)` (48px top and bottom — matches the home composition's 2026-05-17 revision and is the site-wide rhythm at this stage). The composition does **not** introduce a `/404`-specific padding override.
- **Motion**: None at the SiteShell level. Wordmark and nav links are static. Link hover uses the DS's `--easing-link` internally; no site-side override.
- **Content slot**: Nav route list and footer line live in `BaseLayout.astro` / `ShellWrapper.tsx` (the existing site-wide substance carriers). `/404` does not duplicate either; it consumes the same surfaces every other page does.
- **Brand notes**:
  - **`currentRoute={null}`** is load-bearing per spec §9 coordination note: none of the four nav items match `/404`, so none picks up active-state styling. If `<SiteShell>` rejects `null` as a `currentRoute` value, the engineer passes a sentinel value the DS accepts for "no active route" (a non-matching string, e.g. `currentRoute="/404"` — which doesn't match any nav `href` and therefore renders no active state). The composition's intent is "no nav item highlighted"; the engineer picks the mechanical form.
  - The wordmark in the nav is always rendered by `<SiteShell>` via `<Wordmark>`, never a string literal. The wordmark links to `/` (unchanged from every other page). This is itself a salvage path: a visitor who reads no body content at all can still click the wordmark to return home.
  - The footer email link (`mailto:hello@pouk.ai`) is the SiteShell chrome's standing signal that the site is reachable. On `/404` it carries additional weight: it's the only `mailto:` on the page (the body CTA points to `/`, not to email — see Section 2). This is consistent with the spec's framing of `/404` as a *salvage* surface, not a *conversion* surface. The footer email remains because it's chrome, not because it's a conversion path on this specific route.

### Section 2 — `<Hero>` (the dead-end statement and salvage CTA)

- **DS primitive(s)**: `<Hero>` (molecule) from `@poukai-inc/ui@0.9.0`, with one DS atom slotted in:
  - A `<Button asChild size="compact" variant="default"><a href="/">…</a></Button>` in the `cta` slot. **`size="compact"`** is the editorial-restraint rung in the DS's button height ladder (40px min, "quieter than the page default" per `llms-full.txt`). See "Brand notes" for the size justification.
  - **No `<StatusBadge>` in the `status` slot.** Per spec §0 Q4 and AC line 118, D-13 locks `<StatusBadge>` to `/` only. The `status` slot stays empty.
  - **No `illustration` slot.** Per spec §0 Q7 and §10 out-of-scope, `/404` is v1-register only; no figural imagery, no engraving. The composition explicitly does not consume the home composition's pending Pouākai illustration even when the asset eventually lands.
  - **No eyebrow** in this composition's recommendation — see Brand notes for why. (Spec §4 IA item 2 marks the eyebrow optional; spec §5 lets the content-drafter drop it "if `<h1>` reads cleaner without it.")
- **Props (substantive)**:
  ```
  <Hero
    size="intimate"                           // composition decision — see Brand notes
    align="center"                            // composition decision — see Brand notes
    entrance={undefined}                      // composition decision — no motion on /404
    title={<>Draft: This page doesn't exist.</>}    // content-drafter authors final copy
    lede={<>Draft: Try the nav above, or head back to the homepage.</>}
    cta={
      <Button asChild size="compact">
        <a href="/">Draft: pouk.ai</a>
      </Button>
    }
    // status: NOT set — D-13 locks StatusBadge to / only.
    // illustration: NOT set — v1 register, no figural imagery.
    // eyebrow: NOT set — composition recommends a titled Hero without eyebrow.
  />
  ```
- **Layout / spacing**: Internal Hero rhythm — `title → lede → cta` — is owned by `<Hero>` in the DS. At `size="intimate"`, the DS applies the tighter rhythm shipped in 0.7.1 (status→title `--space-3`, title→lede `--space-6` desktop / `--space-4` mobile). The site does **not** override any internal Hero spacing. Hero text column width is capped at `--hero-max` (38rem / 608px) by the DS. The composition does not widen the text column.
- **Motion**:
  - **No `entrance="stagger"`.** This is a deliberate composition decision. Justification under Brand notes; restated here: a staggered reveal on an error page reads as performance — the page is asking the visitor to wait for an apology theatre to complete before they can act. Salvage surfaces want *instant resolution*, not editorial pacing. Static first-paint is the recommended default per spec §8 AC ("the page is short enough that a static first-paint is the recommended default").
  - Link hover uses the DS's `--easing-link` and `--dur-fast` via `<Button>`-internal styling. No site-side override.
  - `prefers-reduced-motion: reduce` is trivially satisfied — no animation to gate. The DS's `:root !important` block in `tokens.css` would handle any future motion addition; this composition does not need to instruct the engineer on motion gating because there is no motion to gate.
- **Content slot**: Body content is **hardcoded in `src/pages/404.astro`**, not driven by a JSON file. Per spec §6 ("`/404` reads no `src/content/*.json` file — its body is hardcoded in the page template (single-author, 3–4 line page; promoting to JSON earns no maintenance benefit)"). The title, lede, and CTA label all source from the approved content draft once it lands.
- **Brand notes**:
  - **`size="intimate"` (not `size="display"`).** The spec authorizes either; this composition recommends `intimate` for three reasons:
    1. **Brand-register match to the surface's job.** `display` is the DS's landing-page register — Hero title clamp 36–68px. `/404` is not a landing page; it's a salvage page. The visitor's intent is not to be impressed by the brand; it's to figure out where they are and where to go next. A 68px-at-desktop title shouting "This page doesn't exist." reads as the page taking itself too seriously.
    2. **Internal-rhythm match to the page's short body.** At ≤50 words of body content, the larger `display` clamp pulls the lede and CTA further from the title than the content density warrants. `intimate`'s tighter rhythm (status→title `--space-3`, title→lede `--space-6`) reads as a single tight statement, which matches the page's "one short statement that names the dead end" intent.
    3. **Consistency with `/`**'s current direction. The home composition adopted `size="intimate"` per the 2026-05-17 revision (D-17 / amendment §4.2 Lever A). `/404` adopting the same Hero density preserves brand-voice continuity: the salvage surface holds the same register as the doorway. A `/404` at `display` while `/` is at `intimate` would read as register-mismatch — the error page louder than the landing page.
  - **`align="center"` (not the DS default).** This is the composition's vertical-positioning call. The spec leaves it open; this composition recommends `align="center"` for two reasons:
    1. **Compositional centering matches the page's single-block intent.** The page is one Hero, one viewport tall. Left-aligning the Hero (the home composition's posture) reads as "the first block of a longer page" — implying more content below. Center-aligning reads as "this is the page." On a salvage surface, the second reading is the correct one.
    2. **DS contract permits.** The DS's `llms-full.txt` Hero rules say `align="center"` is for "standalone landing contexts only." `/404` is a standalone context — single Hero, no other body sections, no scroll target. It qualifies. (If `<SiteShell>`'s page-content wrapper does not vertically center the Hero in the available viewport — i.e., the content sits at the top of the content area regardless of `align="center"` on the Hero itself — the composition accepts top-of-flow positioning as the fallback. The spec's intent is "the entire content sits above the fold" (§4), not "vertically centered in viewport." Horizontal centering of the Hero's text column via `align="center"` is the brand-shape decision; vertical positioning is a layout consequence the engineer accepts as DS-default.)
  - **No `entrance="stagger"`.** Justified above under Motion. Restated: salvage surfaces want instant resolution. The home composition consumes `entrance="stagger"` because `/` is an editorial doorway that earns the ~1.05s reveal; `/404` is not earning a reveal — the visitor is already mid-frustration with a routing failure, and 1.05s of staggered fade-in reads as the page slow-walking the apology. Static first-paint serves the visitor's intent.
  - **No eyebrow.** Spec §4 IA marks the eyebrow optional and spec §5 lets the content-drafter drop it. This composition recommends dropping it because:
    1. **`<h1>` carries the load.** A short declarative title ("This page doesn't exist.") names the dead end completely. An eyebrow like "404" or "Not found" duplicates the title's semantic content without adding routing or register information.
    2. **Body word-count is constrained.** Spec caps total body content at ≤50 words. Burning 1–4 of those words on an eyebrow that duplicates the title is a poor trade against a more breathable lede or a less hedged CTA label.
    3. **Eyebrow precedent across the site is structural-label, not status-label.** Eyebrows like "Role 01" (RoleCard) and "Principle i." (Principle) encode structural position. `/404` has no structural position to encode — it is one of one. An eyebrow here would be the page's only non-structural eyebrow on the site, which is itself a register drift.
    The content-drafter retains the final call per spec §5. If they author an eyebrow that earns its place, this composition does not block it. The recommendation is "drop unless it earns its place."
  - **`<Button size="compact">` on the Hero CTA** (not the DS default `size="md"` and not the home composition's `size="sm"`). Justification:
    - `size="md"` (44px) is the DS's page-default CTA — visually loud, AAA tap target. On `/404` at `<Hero size="intimate">`, a 44px button reads heavier than the title and pulls the visual hierarchy toward "click this CTA" rather than "read the line, then act." Wrong register for a salvage page where the SiteShell nav is the more-likely re-entry path.
    - `size="sm"` (32px) is the home composition's pairing-default for `<Hero size="intimate">`. On `/`, `sm` works because the page also carries a `<StatusBadge>` above the title which establishes a small-element register at the top of the Hero. `/404` has no `<StatusBadge>` — there's nothing else small in the Hero composition for a 32px button to read against. At `sm`, the CTA risks reading as a secondary affordance rather than the page's single salvage path.
    - `size="compact"` (40px, "editorial CTA — quieter than the page default" per `llms-full.txt`) is the proportional middle — small enough to honor the page's restraint, large enough to read as a primary affordance, and matches the DS-published rung that exists exactly for editorial-restrained register. `size="compact"` shipped in 0.9.0 per the DS spec (the DS file references `--btn-h-compact: 40px` and `size="compact"` in the Button rules); the composition consumes the published rung, not a workaround.
    - **Trade-off**: at `compact` (40px), WCAG 2.5.5 AAA tap target (44px) fails; AA (24px) passes comfortably. This matches the home composition's documented trade-off and the DS's own "AA tiers" framing for `sm` and `compact`. Spec §8 quality ACs require accessibility-=-100 on Lighthouse, which evaluates against AA, not AAA. The composition holds.
  - **`variant="default"` (not `variant="primary"`).** The DS rules permit one `variant="primary"` per visual section. The home composition ships with zero primary buttons (the Hero CTA is default-variant). `/404` matches this posture: the CTA is a salvage handoff, not a sales primary. Default-variant carries the right register.
  - **CTA label is content-drafter's call.** The placeholder `Draft: pouk.ai` is one of the spec's example shapes (§5). The drafter may choose any of the spec-authorized shapes (`Back to pouk.ai` / `Home` / `pouk.ai`) or author a different brand-voice-declarative phrase. The composition's only structural requirement is that the label is a single anchor word/phrase, not a conditional sentence (spec §5 differentiation against `/principles` and `/about` v2 end-CTAs).
  - **The Hero is the only `<Hero>` on the page** (DS rule). The Hero CTA is the only `<Button>` on the page. The page carries zero `<StatusBadge>` instances. The page carries one `<h1>` (the Hero title). All DS per-page caps trivially satisfied.
  - **No hydration.** `<Hero>` (and the wrapping `ShellWrapper.tsx`) renders as static HTML at build time. No `client:*` directive. Zero React runtime ships on `/404`. Spec §8 AC ("No new client-side JS shipped on `/404`") satisfied.

### Section 3 — Body link list (**new — rev 2 founder override**)

A 3-row editorial list of important routes with one-line descriptors. Sits below the Hero CTA, above the SiteShell footer chrome. Reads as a *footnoted list* or *sub-footer-style link group* — explicitly NOT a card grid, NOT a button row, NOT a large-type pull-quote treatment.

- **DS primitive(s)**: **None directly.** No `<RoleCard>`, no `<Stat>`, no `<Principle>`, no `<Statement>`. Justification:
  - `<RoleCard>` is too card-shaped (recessed `--surface` fill, `--radius-3` rounded container, eyebrow + icon well) and the DS rule "min 2 cards per use; max 6 per grid" implies a grid register. Wrong shape for a footnote-register list.
  - `<Stat>` is for numerical display. No numerals here.
  - `<Principle>` carries lowercase Roman numerals and editorial-bookend register. Pulling it onto `/404` would steal `/principles`'s vocabulary.
  - `<Statement>` (shipping at 0.11.0) could plausibly fit; this composition explicitly does **not** consume it (see §0 Assumptions). The spec is on 0.9.0; pre-empting a 0.11.0 primitive on a salvage surface is wrong precedent (first consumer of a new DS primitive should be a prominent surface that earns the register, not an error page).
  - **The composition uses plain semantic HTML** (`<nav>` wrapper + `<ul>` + `<li>` rows, each `<li>` containing an `<a href>` + a `<span>` descriptor). Site-side wrapper component, no DS molecule. Classed against the DS token surface (`--font-serif` for the label, `--font-sans` + `--fg-muted` for the descriptor, `--space-N` for row gaps). This is consistent with how `/principles` end-CTA and `/about` v2 sub-footer-register surfaces compose — substance carriers, not DS molecules.
- **Semantic structure**:
  ```html
  <nav aria-label="Suggested routes" class="not-found-suggestions">
    <ul role="list">
      <li>
        <a href="/why-ai">
          <span class="label">Why AI</span>
          <span class="descriptor">Draft: Why most AI projects fail.</span>
        </a>
      </li>
      <li>
        <a href="/roles">
          <span class="label">Roles</span>
          <span class="descriptor">Draft: How we engage — builder, automator, advisor.</span>
        </a>
      </li>
      <li>
        <a href="/principles">
          <span class="label">Principles</span>
          <span class="descriptor">Draft: How we work and what we won't do.</span>
        </a>
      </li>
    </ul>
  </nav>
  ```
- **Route picks (3 of the 4 nav routes)**: `Why AI`, `Roles`, `Principles`. **`About` is intentionally omitted** for two reasons:
  1. **Descriptor-led list register prefers 3 over 4 rows.** Three is the editorial-list canonical count (cf. the home composition's three-sentence lede, `/about` v2's three-paragraph statement). Four rows starts to read as nav-duplicate; three reads as editorial selection. The composition is "the three most important destinations from here," not "all four nav destinations re-listed."
  2. **`/about` v2 is its own destination class.** `/about` v2 is the highest-register editorial page on the site (orange band, display-scale type, portrait). Listing it third or fourth in a salvage-page sub-footer flattens its register. The SiteShell nav already exposes `About` via every page's header and footer; the visitor reaches `/about` through nav, not through a `/404` descriptor.
  - **If the founder prefers 4 routes** (carrying `About`): the composition holds — list extends to 4 rows, ≤6 words per descriptor remains the cap, total list word-count rises to ~24 words (still within the ≤75 page-total cap). Default if no answer: **3 routes** (`Why AI`, `Roles`, `Principles`).
- **Typographic treatment (the composition's verdict)**: **Each row is two-tier — label in `--font-sans` (Geist) at `--fs-body` (17–19px), descriptor in `--font-sans` at `--fs-meta` (14px) `--fg-muted`.** Justification (the call — see also §7 Q5):
  - **Not Instrument Serif italic on the label.** Instrument Serif is the DS's display face — used for `<h1>`, `<h2>`, RoleCard titles, Principle titles, Stat numerals. It signals **editorial weight**. Putting it on the label of a salvage-page sub-footer list would either (a) elevate the list above the Hero's `<h1>` (visual hierarchy break — the Hero title is supposed to be the page's only display-register element) or (b) reduce the Instrument Serif scale to body-size, which strips the face of its intended purpose. The DS's `llms-full.txt` is explicit: "Used for h1, h2, RoleCard titles, Principle titles, FailureMode titles, Stat numeral. **NEVER use for UI chrome.**" A sub-footer link list is UI chrome, not editorial heading text.
  - **Geist (`--font-sans`) at `--fs-body` for the label.** Reads as a *link*, not as a *title*. Matches the SiteShell nav's typographic register (Geist sans). Distinguishes the route label from the Hero `<h1>` (which is Instrument Serif at `--fs-tagline-intimate`). Hierarchy: Hero `<h1>` (display Instrument Serif) > link list label (body Geist) > link list descriptor (meta Geist muted) > footer chrome (meta Geist muted). Clean four-step hierarchy on `--bg`, no surface alternation, no display-scale doubling.
  - **`--fs-meta` (14px) `--fg-muted` for the descriptor.** The DS reserves `--fg-muted` for "captions, footer copy, eyebrow labels, lede paragraphs, hiredBy lines, Stat provenance." A descriptor under a route label is caption-class — it tells the reader what the route is about without re-stating the page's marketing copy. The 14px + muted color makes the descriptor recede behind the label visually; the label is the affordance, the descriptor is the reason-to-click.
  - **Layout per row**: inline on desktop (`label` then `descriptor` separated by `--space-3` / 12px, label sized to its own width, descriptor flowing into remaining width up to `--hero-max`) and stacked on mobile below 480px (`label` on its own line, `descriptor` underneath at `--space-1` / 4px below). Site-side CSS, not DS-internal.
  - **Trade-off acknowledged**: a single-tier list (label only, no descriptor) would be even quieter — closer to the SiteShell nav itself in register. The founder's brief explicitly asks for "route label + short descriptor in muted body," which is two-tier. The composition follows the brief. If the founder prefers single-tier after seeing the rendered list, the composition collapses to label-only with no other clause changes — see §7 Q5.
- **Hairline rule above the list (the composition's verdict)**: **Yes — a single `--hairline` (1px `#D2D2D7`) horizontal rule sits between the Hero CTA and the link list, with `--space-12` (48px) of vertical space above the rule and `--space-8` (32px) below.** Justification:
  - **The list earns a separator.** Without a hairline, the list reads as "fourth element of the Hero" — visually attached to the CTA, semantically blended into the page's primary block. A hairline establishes that the list is a *distinct unit* (a quieter, supplementary register below the Hero, not part of it). This matches the DS rule "Dividers, 1px rules, borders. Always applied as a border/border-top/border-bottom, NEVER as a background. Never render thicker than `--hairline-w: 1px`."
  - **`--space-12` above the rule + `--space-8` below gives the list breathing room without making the page feel sectioned.** Larger gap above (matches the `.site-page` content padding) signals "the Hero ends here"; smaller gap below (`--space-8` = 32px) signals "the list starts immediately, no formal section header." The asymmetry is deliberate.
  - **Trade-off acknowledged**: a hairline-free composition would be even more restrained. The argument against the hairline is "another rule on a salvage page is decoration." The argument for is "the list needs to read as supplementary, not as Hero-tail." This composition picks the hairline; the trade-off is visible to Arian in §7 Q6.
  - **Render width of the rule**: `max-width: var(--hero-max)` (38rem / 608px), centered with the Hero text column. The rule does **not** span `--content-max` (64rem / 1024px) — a full-width rule would read as a section divider, which is too loud for a footnote-register list.
- **Layout / spacing**:
  - **Container width**: `max-width: var(--hero-max)` (38rem / 608px), matching the Hero text column. The list and the Hero share the same column axis — visually anchored to the same vertical line.
  - **Container horizontal alignment**: matches the Hero's `align="center"` decision — the list's container is also horizontally centered in the `.site-page` content area. If Arian flips the Hero to left-aligned (Section 2 §7 Q2), the list flips with it; alignment is shared.
  - **Row spacing**: `--space-4` (16px) gap between rows. Justification: `--space-3` (12px) is too tight — descriptor and next row's label start to compete for the same eye-line. `--space-6` (24px) is too loose — rows start to read as separate cards. `--space-4` is the published token between them.
  - **Below the list**: `.site-page` bottom padding `--space-12` (48px) before SiteShell footer chrome — unchanged from rev 1.
- **Motion**:
  - **No entrance animation on the list.** Same rationale as the Hero (Section 2 — salvage surfaces want instant resolution). Static first-paint.
  - **Link hover transitions** — site-side CSS on `<a:hover>` and `<a:focus-visible>`, using `--dur-fast` (180ms) and `--easing-link`. The descriptor's `--fg-muted` text transitions to `--fg` on hover (subtle "wake up" — caption darkens to body register). The label's underline grows on hover per the DS link convention. Both gated by `prefers-reduced-motion: reduce` via the DS's `:root !important` block (`tokens.css`). The composition does not need to author the gate — the DS-published reduced-motion declaration handles all `transition` and `animation` properties site-wide.
  - **No scroll-triggered effects.** No `IntersectionObserver`, no parallax. Zero JS.
- **Content slot**: List rows are **hardcoded in `src/pages/404.astro`** alongside the Hero substance. Per spec §6, `/404` is single-author and reads no `src/content/*.json` — promoting the suggestions list to JSON would earn no maintenance benefit at 3 rows. The descriptor text sources from the content draft once it lands.
- **Brand notes**:
  - **`<nav aria-label="Suggested routes">` is the semantic wrapper.** Not a `<section>`, not a bare `<ul>`. The `<nav>` element with an explicit `aria-label` distinguishes this list from the SiteShell's primary `<nav aria-label="Primary">` (the DS's `navLabel` default per `llms-full.txt`) so screen-reader users can disambiguate the two nav surfaces by landmark name. The DS's `<SiteShell>` is `min-height: 100dvh` and the primary nav is keyboard-reachable via skip-link convention; this secondary nav has no skip-link affordance and is reachable via DOM order only.
  - **`role="list"` on the `<ul>`**: explicit role override to preserve list semantics in case site CSS strips the default list rendering (`list-style: none` is required to drop the bullet — without `role="list"`, some screen readers omit list semantics when `list-style: none` is applied). Belt-and-suspenders for axe-clean compliance per the user's brief.
  - **No `<h2>` on the list** — spec AC line 116 requires "No `<h2>` renders on the page." The list's `aria-label` carries the semantic name; no visible heading rises above the rows. If a visible heading is requested in a future revision (e.g., a small "Try instead:" eyebrow above the list), it would compose as a `<p>` with `--fs-micro` uppercase-tracked styling, **not** as an `<h2>` — preserves the one-`<h1>`-zero-`<h2>` AC.
  - **The list complements the SiteShell nav, does not duplicate it.** The SiteShell nav surfaces *all four canonical routes* as label-only links in chrome register. This list surfaces *three* of them with descriptor-led editorial register in body register. Different shape, different register, different reading order. The PM's §0 row 3 concern ("would duplicate the nav and add visual weight") is mitigated by the two-tier descriptor register — the list reads as "here are the three places worth going next, with reasons," not as "the same four nav links again."
  - **`mailto:hello@pouk.ai` does NOT appear in the list.** The list is route-internal only. Spec AC line 120 forbids `mailto:` on the body CTA; the same restraint extends to the new list. The salvage surface's job is re-entry to the canonical funnel, not direct conversion. The footer chrome's `mailto:` line is the only `mailto:` on the page.
  - **Hover/focus state for keyboard users**: `<a:focus-visible>` shows a 2px `--accent` (`#0071E3`) outline at `--radius-1` (2px) per the DS's focus convention. The composition does not author the focus style; site CSS reuses the DS's published focus token pattern.

## 4. Motion choreography (page-level)

The page ships **zero JavaScript and zero CSS keyframe animations** on its own surfaces. The only motion on the page is link hover transitions (DS-owned where the underline lives in a DS primitive; site-side CSS where the link is in the new Section 3 list):

- **No Hero entrance animation.** `<Hero entrance>` is not set. Justification in Section 2 brand notes: salvage surfaces want instant resolution, not editorial pacing.
- **No link-list entrance animation.** The list does not stagger or fade in. Justification: same as Hero — instant resolution. Adding a stagger to the list while the Hero is static would invert the visual hierarchy (the supplementary unit animates, the primary unit doesn't) and read as wrong.
- **No StatusBadge pulse** (no StatusBadge on the page).
- **No illustration motion** (no illustration on the page).
- **Link hover transitions** — `--dur-fast` (180ms) and `--easing-link`. Applies to:
  - (a) nav links in `<SiteShell>` — DS-internal.
  - (b) the Hero CTA's `<Button asChild><a>` underline — DS-internal.
  - (c) the footer email link in the SiteShell footer — DS-internal.
  - (d) **the new body link list's `<a>` anchors — site-side CSS.** Two transitions per row: descriptor color `--fg-muted → --fg` on hover (subtle "wake up"), and label underline grow per the DS link convention. Both use `--dur-fast` + `--easing-link` for consistency with DS-published link motion.
- **All link hover transitions disabled under `prefers-reduced-motion: reduce`** via the DS's `:root !important` block in `tokens.css`. The site-side CSS for the list's link transitions is gated by the same global declaration — no per-component `@media (prefers-reduced-motion)` rule needed.

**Fires on initial render**: nothing on the page's own surfaces. (DS-internal first-paint styling for `<Wordmark>`, `<SiteShell>`, `<Hero>` applies; none of those are page-composition motion.)

**Fires on scroll**: nothing. The page is essentially-one-viewport tall at desktop; mobile may scroll to reveal the list bottom, but no scroll-triggered reveal or transform is consumed. Scroll-triggered animation has no payoff and would force a `client:*` directive, breaking the zero-JS contract.

**Fires never (locked out by this composition)**: Hero entrance staggers, link-list entrance staggers, illustration animations, scroll-triggered reveals, parallax, intersection-observer-driven effects, any JS-required motion. Any future urge to add motion to `/404` is a composition revision and must clear (a) the spec's zero-JS AC, (b) the DS's `prefers-reduced-motion` gate, (c) the salvage-surface intent ("instant resolution, not editorial pacing").

**`prefers-reduced-motion: reduce` behavior**: trivially satisfied at the keyframe layer (no keyframe animations exist). Link hover transitions are reduced/disabled via the DS's `:root !important` block automatically.
